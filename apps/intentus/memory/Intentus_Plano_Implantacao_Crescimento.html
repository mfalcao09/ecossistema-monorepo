1. O que Falta — Ferramentas, Tecnologias, Segurança e Inputs

Diagnóstico: O Intentus hoje tem 11 módulos desenhados e Power BI como BI primário, focados em incorporação. Para se tornar a plataforma do ecossistema imobiliário, existem gaps significativos em 4 camadas: tecnologia/stack, segurança/compliance, ferramentas/funcionalidades, e dados/inputs necessários.

Gaps Críticos — Tecnologia & Stack

🔴 Arquitetura base inexistente

O Intentus não tem codebase. Precisa de toda a infraestrutura fundacional antes de qualquer módulo funcional.

  • Multi-tenancy framework: Isolamento de dados por cliente é obrigatório para SaaS B2B. Duas abordagens: schema-per-tenant (mais seguro, mais caro) ou row-level security (mais eficiente, requer rigor)
  • IAM (Identity & Access Management): RBAC (Role-Based Access Control) com granularidade por módulo, vertical e ação. Multi-fator obrigatório para verticais de capital markets
  • Event bus / Message broker: Kafka ou RabbitMQ para comunicação assíncrona entre módulos. Essencial para que incorporação "converse" com locação sem acoplamento
  • API Gateway: Kong, AWS API Gateway ou custom com rate limiting, versionamento, autenticação por API key e OAuth2
  • CI/CD Pipeline: GitHub Actions ou GitLab CI com testes automatizados, staging, e deploy automatizado para Kubernetes ou ECS
  • Monitoramento e Observabilidade: Datadog ou Grafana+Prometheus para métricas de infra; Sentry para error tracking; OpenTelemetry para tracing distribuído

🔴 BI Nativo — substituir Power BI

Power BI como BI primário é um dos maiores gaps. Cria dependência de licença Microsoft, experiência fragmentada e custo por usuário.

  • Opção 1 — Embedded analytics (Metabase/Apache Superset): Open-source, embedável no Intentus, customizável. Custo: R$0 (self-hosted) ou R$500-2000/mês (cloud)
  • Opção 2 — Custom dashboards (React + Recharts/D3.js): 100% controle, UX premium, mas exige mais dev time. Ideal para dashboards core
  • Opção 3 — Cube.js como camada semântica: Headless BI que serve dados para qualquer frontend. Melhor equilíbrio entre flexibilidade e velocidade
  • Recomendação: Cube.js (camada de dados) + custom React dashboards (UX premium) + Metabase (relatórios ad-hoc para power users)

🔴 Motor de IA / ML Pipeline

IA como diferencial precisa de infraestrutura dedicada, não apenas chamadas a APIs externas.

  • ML Pipeline: MLflow ou Kubeflow para treinamento, versionamento e deploy de modelos
  • Feature Store: Armazenamento de features para modelos de precificação, score de risco e recomendação
  • LLM Integration: Anthropic Claude ou OpenAI para geração de documentos, análise de contratos, due diligence assistida. Via API com fallback entre providers
  • Vector Database: Pinecone ou Weaviate para RAG (Retrieval Augmented Generation) sobre documentos imobiliários
  • Modelos core necessários: Precificação comparativa (regressão), Score de risco/inadimplência (classificação), Detecção de anomalias em transações (compliance), Extração de dados de documentos (OCR + NLP)

🟠 Document Management System (DMS)

Cada vertical gera volumes enormes de documentos. Precisa de GED (Gestão Eletrônica de Documentos) robusta.

  • Storage: AWS S3 com lifecycle policies (documentos ativos vs. arquivo morto)
  • OCR: Textract (AWS) ou Google Document AI para extração automática de dados de contratos, matrículas, certidões
  • Assinatura eletrônica: Integração com Clicksign ou DocuSign (ICP-Brasil para validade jurídica plena)
  • Versionamento: Controle de versão de contratos com diff visual
  • Templates: Engine de geração de documentos a partir de templates (minutas, relatórios CVM, laudos)

🟠 BaaS (Banking as a Service) Engine

Funcionalidades financeiras embedded são diferencial competitivo massivo.

  • Emissão de boletos: Via API de bank-as-a-service (Celcoin, Asaas, ou Pagarme)
  • Conciliação bancária automatizada: Matching automático entre pagamentos e contratos via OFX/CNAB
  • Split de pagamentos: Divisão automática entre proprietário, administradora, fundo de reserva
  • Pix integrado: QR codes dinâmicos com conciliação em tempo real
  • Conta escrow: Para patrimônio de afetação e operações de incorporação
  • DIMOB / NF-e: Geração automática de declarações fiscais obrigatórias

🟠 Integrações Essenciais (Camada 2)

  • Portais imobiliários: ZAP Imóveis, OLX, VivaReal (API de publicação de anúncios)
  • Cartórios: e-Notariado e Registro de Imóveis Eletrônico (SREI)
  • Bureaus de crédito: Serasa Experian, Boa Vista para análise de inquilinos/compradores
  • Seguradoras: APIs de seguro fiança (Porto Seguro, Tokio Marine)
  • Bancos: APIs bancárias para acompanhamento de financiamento e repasse
  • CVM / B3: Para compliance automatizado de FIIs (vertical Capital Markets)
  • Prefeituras: NF-e, IPTU, certidões (via APIs municipais ou scraping regulado)
  • ERPs: Coexistência com Sienge, Totvs, SAP (não substituição — migração gradual)

🟡 Inputs / Dados Necessários

O Intentus precisa de fontes de dados externas para alimentar IA e benchmarking.

  • Dados de mercado imobiliário: FIPE ZAP (preços), SECOVI (indicadores setoriais), CBIC (construção)
  • Dados macroeconômicos: Selic, IPCA, IGPM, INCC via APIs do BCB e IBGE
  • Dados de FIIs: B3 (cotações, proventos), CVM (informes mensais, anuais)
  • Dados cadastrais: CNPJ via Receita Federal, geolocalização via Google Maps/Here
  • Dados judiciais: APIs de tribunais (TJ, STJ) para tracking de processos em distressed/leilões
  • Dados de leilões: Web scraping estruturado de leiloeiros oficiais (Sato, Zukerman, Mega Leilões)

🟡 Comunicação Multi-Canal (via Nexvy)

  • WhatsApp Business API: Notificações transacionais (boleto gerado, contrato pronto, assembleia)
  • Email transacional: SendGrid ou Amazon SES para automações (réguas de cobrança, lembretes)
  • Push notifications: Firebase Cloud Messaging para app mobile
  • SMS: Backup para notificações críticas (vencimento, convocação de assembleia)
  • In-app notifications: Centro de notificações dentro da plataforma

Gaps de Segurança & Compliance

Por que isso importa: O Intentus vai lidar com dados sensíveis de compradores, inquilinos, cotistas, transações financeiras, e informações reguladas pela CVM. Segurança não é feature — é pré-requisito para vender para gestoras e fundos.

🔴 LGPD — Lei Geral de Proteção de Dados

Obrigatório para qualquer SaaS no Brasil. A ANPD está intensificando fiscalizações.

  • DPO (Encarregado de Dados): Nomear um responsável (pode ser terceirizado inicialmente)
  • Mapeamento de dados: Inventário completo de quais dados pessoais são coletados, onde armazenados, por quanto tempo, com quem compartilhados
  • Consentimento: Mecanismo de coleta e gestão de consentimento (opt-in explícito)
  • Direitos do titular: Portal/API para que titulares exerçam direitos (acesso, correção, exclusão, portabilidade)
  • Privacy by Design: Pseudonimização de dados sensíveis, minimização de coleta, retenção limitada
  • RIPD: Relatório de Impacto à Proteção de Dados para operações de alto risco
  • Custo estimado: R$ 15.000-40.000 para implementação inicial + R$ 2.000-5.000/mês manutenção

🔴 SOC 2 Type II

Certificação que gestoras e fundos exigem para contratar SaaS. Sem ela, o ciclo de vendas para Capital Markets e Gestão de Ativos será 3-5x mais longo.

  • Trust Service Criteria: Security (obrigatório), Availability, Processing Integrity, Confidentiality, Privacy
  • Cronograma: 6-12 meses da decisão até o relatório Type II
  • Etapas: Gap analysis → Remediação → Implementação de controles → Audit window (3-12 meses) → Relatório
  • Controles técnicos: Encryption at rest (AES-256) e in transit (TLS 1.3), MFA para todos os acessos, Logging centralizado, Vulnerability scanning, Incident response plan
  • Custo estimado: US$ 30.000-50.000 (R$ 165.000-275.000) para startup, incluindo ferramentas de compliance automation (Vanta, Drata ou Sprinto)
  • Timing ideal: Iniciar preparação na Fase 2 (mês 9), obter Type I na Fase 3, Type II na Fase 4

🟠 Segurança da Aplicação

  • OWASP Top 10: Proteção contra SQL injection, XSS, CSRF, broken authentication — testes automatizados no CI/CD
  • WAF (Web Application Firewall): AWS WAF ou Cloudflare para proteção de borda
  • Rate Limiting & DDoS: Proteção contra abuso de APIs e ataques volumétricos
  • Secret Management: AWS Secrets Manager ou HashiCorp Vault para chaves, tokens, credentials
  • Penetration Testing: Pentests anuais obrigatórios, idealmente semestrais. Custo: R$ 15.000-40.000/teste
  • Dependency scanning: Snyk ou Dependabot para vulnerabilidades em dependências

🟡 Backup, Disaster Recovery & SLA

  • Backup: Automatizado, diário, com retenção de 30 dias mínimo. Cross-region para disaster recovery
  • RPO (Recovery Point Objective): Máximo 1 hora de dados perdidos em caso de falha
  • RTO (Recovery Time Objective): Máximo 4 horas para restauração completa
  • SLA target: 99.9% uptime (8.7h de downtime/ano) — necessário para vender para gestoras
  • Multi-AZ: Deploy em múltiplas availability zones (AWS sa-east-1 com failover)
  • Incident Response Plan: Documentado, testado, com runbooks para cenários comuns

Veredicto: Incorporação é a primeira vertical — mas Locação vem logo atrás

Marcelo, sua intuição sobre Locação está correta — é uma vertical excelente e com mercado gigantesco (40.000+ administradoras). Porém, Incorporação tem 3 vantagens decisivas para ser a primeira:

  • Dogfooding com o Splendori: Você tem um empreendimento real para testar. Isso é insubstituível. Nenhum SaaS nasce bem sem ser usado internamente primeiro.
  • 11 módulos já desenhados: O escopo de incorporação já está mapeado no Intentus atual. Locação precisaria de um redesign quase completo.
  • Ticket maior que locação: R$997-2.497/mês/empreendimento vs R$3-5/unidade. Para atingir R$50k MRR, precisa de 20-50 clientes em incorporação vs 100+ em locação.

Sequência recomendada: Incorporação (Fase 1) → Locação (Fase 2) → Gestão de Ativos + Capital Markets (Fase 3) → Distressed + Leilões (Fase 4)

Isso aproveita a base de incorporação: quando o empreendimento termina, as unidades não vendidas vão para locação. Os portfólios locados alimentam gestão de ativos. É uma narrativa de crescimento natural.

Por que NÃO começar pela Locação (apesar do mercado maior)?

  • Concorrência mais madura: Kenlo tem 3.000+ clientes e 400k contratos. Entrar num mercado com incumbente forte sem diferencial claro é mais arriscado.
  • Ticket baixo demais para MVP: A R$3-5/unidade, você precisa de volume altíssimo para sustentar o negócio nos primeiros meses.
  • Sem dogfooding: Você não administra imóveis para locação. Sem experiência prática, o feedback loop é mais lento.
  • O diferencial do Intentus em locação aparece quando já existe a base de incorporação: "Seu empreendimento foi entregue? Agora administre as unidades locadas no mesmo sistema." Esse é o upsell natural.

2. Plano de Implantação de Módulos — Passo a Passo

Filosofia de implementação: Começar com monólito modular (não microserviços). Startups que começam com microserviços gastam 60-80% do esforço em infraestrutura, não em produto. Monólito modular permite lançar rápido e migrar para microserviços quando tiver tração (50+ clientes).

  • CNAE principal: 6201-5/01 (Desenvolvimento de programas de computador sob encomenda) e 6203-1/00 (Desenvolvimento e licenciamento de programas de computador customizáveis)

  • Vesting: Oferecer equity pool (10-15%) para CTO e key engineers com cliff de 12 meses e vesting de 48 meses

Governança Interna

  • Board mensal: Marcelo + CTO + advisor externo (ideal: alguém do mercado imobiliário ou de SaaS B2B)
  • Sprint reviews: Quinzenais com todo o time (produto + business)
  • OKRs trimestrais: Alinhamento de metas por squad (tech, produto, comercial)
  • 1:1s semanais: CTO→devs, Marcelo→CTO/vendas/CS
  • All-hands mensal: Transparência total sobre métricas, finanças e roadmap

Estrutura Organizacional por Fase

Fase 1: Startup Squad (8-10 pessoas)

Marcelo como CEO/CPO direto. CTO lidera tech. Sem camadas de gestão. Todo mundo codando ou vendendo.

Marcelo (CEO/CPO) → CTO → Squad único de produto

Fase 2: Dois Squads (14-17 pessoas)

Split por vertical: Squad Incorporação (manutenção + evolução) e Squad Locação (construção + lançamento). CS e vendas como funções separadas.

Marcelo (CEO) → CTO (2 squads tech) + Head Comercial (vendas + CS) + Marketing

Fase 3-4: Departamentos (22-30 pessoas)

Squads por vertical/domínio. Especialistas (IA, data, security) como shared services. Vendas enterprise com ciclo longo para gestoras.

Marcelo (CEO) → CTO (4 squads + platform) + VP Comercial (vendas + CS + marketing) + Head de Compliance
