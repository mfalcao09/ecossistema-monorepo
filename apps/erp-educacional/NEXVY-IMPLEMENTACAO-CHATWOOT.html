Banco de Dados

Tabelas novas no mesmo Supabase do ERP. Sem novo projeto.

-- Tabelas novas no Supabase existente do ERP -- Fase 1: SEM account_id (single-tenant = FIC é implícito) -- Fase 2: ADD COLUMN account_id + RLS em todas atendimento_inboxes -- canais: WhatsApp, Instagram, Messenger atendimento_contacts -- contatos: alunos, interessados, responsáveis atendimento_conversations -- conversas (status: open/resolved/pending) atendimento_messages -- mensagens (alto volume, indexado) atendimento_labels -- etiquetas: matrícula, financeiro, acadêmico atendimento_automation_rules -- regras IF/THEN em JSONB atendimento_agents -- funcionários da FIC que atendem

Preparado para a Fase 2 desde o início

-- Fase 2: adicionar account_id em todas as tabelas (uma migration) ALTER TABLE atendimento_conversations ADD COLUMN account_id UUID REFERENCES accounts(id); -- E ativar RLS (a FIC vira account_id = 1) ALTER TABLE atendimento_conversations ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_isolation ON atendimento_conversations USING (account_id IN ( SELECT account_id FROM account_users WHERE user_id = auth.uid() ));

Ao nomear as tabelas com prefixo atendimento_ e já pensar em account_id na estrutura (mesmo que vazio na Fase 1), a migração para multi-tenancy na Fase 2 é uma migration — não uma reescrita.
