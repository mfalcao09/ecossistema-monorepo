## Bug 1: Data de Publicação (-1 dia) — TIMEZONE

### Causa raiz
O Supabase retorna campos `timestamptz` no formato `2024-12-16 00:00:00+00`, mas a função `formatarData()` só tratava datas no formato exato `YYYY-MM-DD` (10 caracteres). O regex `/^\d{4}-\d{2}-\d{2}$/` não matchava o formato longo, então o `new Date()` parseava com timezone UTC e o `toLocaleDateString('pt-BR')` convertia para o fuso local (UTC-3), resultando em 15/12/2024 em vez de 16/12/2024.

### Correção
Trocar o regex exato por extração dos primeiros 10 caracteres:

```typescript
// ANTES (só matchava YYYY-MM-DD exato):
const safe = /^\d{4}-\d{2}-\d{2}$/.test(iso) ? `${iso}T12:00:00` : iso

// DEPOIS (extrai YYYY-MM-DD de qualquer formato):
const match = iso.match(/^(\d{4}-\d{2}-\d{2})/)
const safe = match ? `${match[1]}T12:00:00` : iso
```

### Arquivos corrigidos (4 instâncias de `formatarData`)
1. `src/app/(portal)/PortalPageClient.tsx`
2. `src/app/verificar/[codigo]/page.tsx` (+ `formatarDataHora`)
3. `src/components/portal/DiplomaCardPublico.tsx`
4. `src/lib/portal/validacoes.ts`

### Lição aprendida
**Nunca confiar no formato de data retornado pelo Supabase.** Campos `date` retornam `YYYY-MM-DD`, mas campos `timestamptz` retornam `YYYY-MM-DD HH:MM:SS+00`. Sempre extrair a parte da data com regex, nunca comparar formato exato.

## Bug 2: Forma de Acesso e Data de Ingresso = null — NEXT.JS DATA CACHE

### Sintoma
A API `/api/documentos/verificar/1606.694.c350ea096bac` retornava `data_ingresso: null` e `forma_acesso: null`, mesmo com os dados confirmados no banco via SQL direto.

### Investigação
1. ✅ Dados confirmados na tabela `diplomas` via SQL direto
2. ✅ Tipos TypeScript (`RawDiplomaCompleto`) incluem os campos
3. ✅ Normalização (`normalizarDiplomaCompleto`) mapeia corretamente
4. ✅ Engine (`verificarDocumento`) repassa os campos
5. ✅ Colunas existem no `information_schema` (`data_ingresso: date`, `forma_acesso: varchar`)
6. ✅ Tabela é regular (`relkind: r`), não view
7. ❌ API retorna null — campos presentes mas com valor null

### Causa raiz: Next.js Data Cache
No Next.js App Router (produção), o `fetch()` interno é cacheado por padrão. O Supabase JS client usa `fetch()` internamente para se comunicar com o PostgREST. Quando o diploma foi acessado pela primeira vez (antes de popular `data_ingresso` e `forma_acesso`), o Next.js cacheou a resposta do PostgREST com valores null e nunca mais consultou o banco.

**Nota:** Também foi tentado `NOTIFY pgrst, 'reload schema'` no Supabase, que não resolveu porque o problema NÃO era cache do PostgREST — os campos estavam presentes no JSON (como null), não ausentes.

### Correção (3 arquivos)

**1. Route handler — forçar execução dinâmica:**
```typescript
// src/app/api/documentos/verificar/[codigo]/route.ts
export const dynamic = 'force-dynamic'
```

**2. Supabase clients — desabilitar cache do fetch:**
```typescript
// src/lib/diplomas/buscar-completo.ts
// src/lib/documentos/engine.ts
// src/app/api/portal/consultar-cpf/route.ts
function getAdminClient() {
  return createAdminClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!,
    {
      auth: { autoRefreshToken: false, persistSession: false },
      global: {
        fetch: (url: RequestInfo | URL, init?: RequestInit) =>
          fetch(url, { ...init, cache: 'no-store' }),
      },
    }
  )
}
```

### Lição aprendida
**SEMPRE configurar `cache: 'no-store'` nos Supabase clients server-side em Next.js App Router.** O default do Next.js é cachear `fetch()` em produção, o que é desastroso para dados que mudam (diplomas sendo atualizados, campos sendo populados). O `export const dynamic = 'force-dynamic'` no route handler é uma camada extra de segurança.
