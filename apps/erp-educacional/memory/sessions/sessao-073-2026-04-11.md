## Objetivo
Fix Bug #4 (continuação da série 071-073): identificar e corrigir a causa raiz real do problema dos botões de auditoria XSD que continuavam abrindo o formulário legado vazio.

## Sprint/Epic
- **Sprint 2 / Epic 2.1** — Motor XML polish (fixes da pipeline de auditoria)

## Problema resolvido
Após os fixes das sessões 071 (prop chain + processoId) e 072 (dual-path + sessaoId), os botões de correção da auditoria ainda abriam o formulário legado. Marcelo reportou novamente com a mensagem:

> "Os botões de auditoria continuam me levando para o mesmo lugar, o formulário antigo e vazio. PORRAAAA CLAUDINHO"

**Causa raiz real (encontrada nesta sessão):**

A rota `GET /api/diplomas/[id]` retornava `extracao: null` **hardcoded** — havia inclusive um comentário explícito:

```typescript
extracao: null,    // módulo de extração IA — não implementado nesta fase
```

Isso significava que mesmo com toda a prop chain corretamente conectada (sessões 071+072), o estado `extracao` no `page.tsx` **sempre era null**. Consequentemente, `sessaoId={extracao?.id}` sempre recebia `undefined` → `BotaoCorrecao` sempre caía no fallback legado `/diploma/processos/{processoId}`.

**Evidência do fallthrough:**
```
page.tsx → setExtracao(d.extracao ?? null) → null
PainelAcoes → sessaoId={extracao?.id} → undefined
BotaoCorrecao → sessaoId falsy → rota legada
```

## Solução
Adição de query real à tabela `extracao_sessoes` na rota `GET /api/diplomas/[id]`:

```typescript
// ── Sessão de extração IA (novo fluxo) ──────────────────────────────────
let extracaoSessao: {
  id: string
  status: string
  confianca_geral: number | null
  campos_faltando: string[] | null
  dados_extraidos: Record<string, unknown> | null
  dados_confirmados: Record<string, unknown> | null
  created_at: string
} | null = null

if (d.processo_id) {
  const { data: extRaw } = await admin
    .from('extracao_sessoes')
    .select('id, status, confianca_geral, campos_faltando, dados_extraidos, dados_confirmados, created_at')
    .eq('processo_id', d.processo_id)
    .order('created_at', { ascending: false })
    .limit(1)
    .maybeSingle()
  if (extRaw) extracaoSessao = extRaw as unknown as NonNullable<typeof extracaoSessao>
}
```

E substituição no response:
```typescript
// antes:
extracao: null,    // módulo de extração IA — não implementado nesta fase
// depois:
extracao: extracaoSessao,
```

**Nota técnica:** O TypeScript exigiu o cast duplo `as unknown as NonNullable<typeof T>` porque `typeof extracaoSessao` inclui `null` na união, causando erro de conversão sem o intermediário `unknown`.

## Fluxo corrigido end-to-end
1. `GET /api/diplomas/[id]` busca `extracao_sessoes WHERE processo_id = diploma.processo_id ORDER BY created_at DESC LIMIT 1`
2. Retorna `extracaoSessao` com `id` UUID real
3. `page.tsx` → `setExtracao(d.extracao ?? null)` popula estado com objeto real
4. `sessaoId={extracao?.id}` → UUID não-nulo para `PainelAcoes`
5. Propaga: `PainelAcoes → PainelAuditoria → GrupoExpandivel → IssueRow → BotaoCorrecao`
6. `BotaoCorrecao` → `sessaoId` truthy → abre `/diploma/processos/novo/revisao/{sessaoId}` ✅

## Lição aprendida
Quando uma correção de prop chain não resolve o problema visível, verificar se a **fonte dos dados** (a rota API) está realmente retornando os dados — não apenas que o componente está preparado para recebê-los. Neste caso 3 sessões foram necessárias para rastrear o bug de baixo para cima:
- S071: prop chain (sem dados)
- S072: roteamento dual (ainda sem dados)  
- S073: **fonte dos dados** (causa raiz real)
