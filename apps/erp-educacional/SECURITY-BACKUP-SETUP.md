## Encrypted Database Backups

### Architecture

The backup system consists of three components:

1. **Supabase Edge Function** (`supabase/functions/backup-criptografado/`)
   - Exports critical database tables as JSON
   - Encrypts data using AES-256-GCM
   - Uploads encrypted backups to Supabase Storage
   - Logs backup metadata to `backup_log` table

2. **Vercel Cron Endpoint** (`src/app/api/cron/backup/route.ts`)
   - Triggered daily at 3AM UTC
   - Calls the Supabase Edge Function
   - Determines backup type (daily/weekly/monthly)
   - Logs execution status

3. **Backup Log Database** (`supabase/migrations/20260326_backup_log.sql`)
   - Tracks all backup operations
   - Stores encryption metadata
   - Records integrity hashes (SHA256)
   - Accessible only to admin users

### Backup Types

- **Daily** (default): Regular incremental backups
- **Weekly** (Sundays): Cumulative week-over-week backups
- **Monthly** (1st of month): Full month snapshots

### Supported Storage Backends

- **Supabase Storage** (recommended): Integrated, secure, accessible
- **Cloudflare R2** (future): For multi-cloud redundancy

### Tables Backed Up

```
Critical business data:
- usuario (users)
- usuario_papeis (user roles)
- papeis (roles/permissions)
- instituicoes (institutions)
- departamentos (departments)
- cursos (courses)
- diplomados (graduates)
- diplomas (diplomas)
- diplomas_assinaturas (signatures)
- historico_escolar (academic history)
- pessoas (people)
- enderecoes (addresses)
- documentos (documents)
- diretores_assinantes (signers)
- assinantes (signers list)
- configuracoes_instituicao (institution config)
- backup_log (backup history)
```

### Encryption Details

**Algorithm**: AES-256-GCM (Advanced Encryption Standard 256-bit with Galois/Counter Mode)

**Components**:
- IV (Initialization Vector): 16-byte random value, base64-encoded
- Auth Tag: Generated by GCM mode for integrity verification
- Key Version: Identifier for key rotation

**Backup Storage Structure**:
```
backups/
тФЬтФАтФА 2026-03-26/
тФВ   тФЬтФАтФА backup-diario-2026-03-26T03-00-00.json.enc
тФВ   тФЬтФАтФА backup-diario-2026-03-27T03-00-00.json.enc
тФВ   тФФтФАтФА ...
тФЬтФАтФА 2026-03-23/
тФВ   тФФтФАтФА backup-semanal-2026-03-23T03-00-00.json.enc
тФФтФАтФА 2026-03-01/
    тФФтФАтФА backup-mensal-2026-03-01T03-00-00.json.enc
```

### Backup Log Schema

The `backup_log` table tracks:

| Column | Type | Description |
|--------|------|-------------|
| `id` | UUID | Primary key |
| `tipo` | TEXT | 'diario', 'semanal', 'mensal', 'manual' |
| `status` | TEXT | 'sucesso', 'erro', 'em_andamento' |
| `tabelas_incluidas` | TEXT[] | List of tables backed up |
| `tamanho_bytes` | BIGINT | Original data size |
| `tamanho_criptografado_bytes` | BIGINT | Encrypted data size |
| `algoritmo_criptografia` | TEXT | 'AES-256-GCM' |
| `versao_chave` | TEXT | Key version (e.g., 'v1') |
| `armazenamento` | TEXT | 'r2', 'supabase_storage', 'local' |
| `caminho_arquivo` | TEXT | Full path to encrypted backup file |
| `hash_sha256` | TEXT | Integrity verification hash |
| `inicio` | TIMESTAMPTZ | Backup start time |
| `fim` | TIMESTAMPTZ | Backup end time |
| `erro_mensagem` | TEXT | Error details if failed |
| `created_at` | TIMESTAMPTZ | Log creation timestamp |

### Row-Level Security (RLS)

All backup log access is restricted to admin users:

```sql
CREATE POLICY "Apenas admins veem logs de backup" ON backup_log
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM usuario_papeis up
      JOIN papeis p ON p.id = up.papel_id
      WHERE up.user_id = auth.uid()
      AND p.slug = 'admin'
    )
  );
```

### Backup Schedule

Cron job configuration in `vercel.json`:

```json
{
  "crons": [
    {
      "path": "/api/cron/backup",
      "schedule": "0 3 * * *"
    }
  ]
}
```

**Schedule**: Daily at 3AM UTC (synchronized with other maintenance tasks)

### Manual Backup Trigger

To trigger a manual backup outside the scheduled time:

```bash
curl -X POST https://your-app.vercel.app/api/cron/backup \
  -H "Authorization: Bearer $CRON_SECRET" \
  -H "Content-Type: application/json" \
  -d '{"tipo": "manual"}'
```

### Backup Verification

Check backup logs via Supabase dashboard:

```sql
SELECT
  id,
  tipo,
  status,
  tamanho_bytes,
  tamanho_criptografado_bytes,
  hash_sha256,
  inicio,
  fim
FROM backup_log
ORDER BY created_at DESC
LIMIT 10;
```

Query by status:

```sql
SELECT * FROM backup_log
WHERE status = 'erro'
ORDER BY created_at DESC;
```

## OWASP ZAP Security Scanning

### What is OWASP ZAP?

OWASP ZAP (Zed Attack Proxy) is an open-source security testing tool that:
- Performs automated vulnerability scanning
- Identifies OWASP Top 10 security issues
- Generates detailed security reports
- Integrates with CI/CD pipelines

### Scan Types

#### 1. Baseline Scan (`.github/workflows/security-scan.yml`)

Lightweight security scan that:
- Crawls the application at normal speed
- Tests for common vulnerabilities
- Runs on every push and pull request
- Takes ~5-10 minutes

Common findings:
- Missing security headers
- Weak SSL/TLS configuration
- Cookie security issues
- XSS vulnerabilities
- CSRF token problems

#### 2. API Scan

Targeted scan for REST/GraphQL APIs:
- Tests authentication & authorization
- Validates input validation
- Checks API-specific security issues
- Uses OpenAPI/Swagger specs if available

#### 3. Scheduled Weekly Scan

Full security audit run every Monday at 4AM UTC:
- More aggressive scanning options
- Deeper vulnerability detection
- Used for comprehensive security review

### Rules Configuration

Rules are configured in `.zap/rules.tsv`:

```tsv
Rule ID | Action | Description
10010   | IGNORE | Cookie No HttpOnly Flag (managed by framework)
10020   | FAIL   | Missing X-Frame-Options Header
40012   | FAIL   | Cross Site Scripting (Reflected)
40018   | FAIL   | SQL Injection
```

**Actions**:
- `IGNORE`: Rule not checked
- `WARN`: Logged as warning, doesn't fail build
- `FAIL`: Blocks deployment if triggered

### Allowlist Configuration

File: `.zap/allowlist.conf`

URLs excluded from scanning:
```
/_next/*         # Next.js framework routes
/_static/*       # Static assets
/api/cron/*      # Cron endpoints (covered separately)
```

## CI/CD Workflows

### Security Scan Workflow

**File**: `.github/workflows/security-scan.yml`

**Trigger**: Every push, PR, and weekly

**Jobs**:

1. **Build and Scan** (main job)
   - Checkout code
   - Install dependencies
   - Build application
   - Start dev server
   - Run ZAP baseline scan
   - Run ZAP API scan
   - Upload HTML report

2. **npm Audit** (dependency check)
   - Check for vulnerable packages
   - Fail on critical issues
   - Generate audit report

3. **Type Check** (code quality)
   - TypeScript compilation
   - ESLint validation

4. **Report Summary** (aggregation)
   - Combines all results
   - Posts to GitHub Actions summary

### Dependency Audit Workflow

**File**: `.github/workflows/npm-audit.yml`

**Trigger**: Daily at 6AM UTC, on `package.json` changes

**Checks**:
- Critical vulnerabilities тЖТ FAIL build
- High vulnerabilities тЖТ WARN
- Auto-generates audit report
- Comments on PRs with summary

### Workflow Status

Check status in GitHub:
```
Actions tab тЖТ Security Scan тЖТ View run details
```

Artifacts available:
- `zap-baseline-report` (HTML)
- `zap-api-report` (HTML)
- `npm-audit-report` (JSON)

## Backup Recovery

### Decrypt Backup Programmatically

```typescript
import { descriptografarBackup } from '@/lib/backup/crypto'

const params = {
  encrypted: backupBuffer,
  iv: 'base64-encoded-iv',
  tag: 'base64-encoded-tag',
  key: process.env.BACKUP_ENCRYPTION_KEY || '',
}

try {
  const decrypted = await descriptografarBackup(params)
  const data = JSON.parse(decrypted.toString())
  console.log('Backup recovered:', data)
} catch (error) {
  console.error('Decryption failed:', error)
}
```

### Manual Recovery Process

1. **Verify backup exists**:
   ```sql
   SELECT * FROM backup_log WHERE status = 'sucesso'
   ORDER BY created_at DESC LIMIT 1;
   ```

2. **Download encrypted file** from Supabase Storage:
   - Navigate to Supabase Dashboard
   - Storage тЖТ backups folder
   - Download `.enc` file

3. **Prepare recovery script** with Node.js:
   ```typescript
   // See decryption example above
   ```

4. **Restore data** to new database:
   ```sql
   INSERT INTO usuario SELECT * FROM imported_data.usuario;
   INSERT INTO diplomas SELECT * FROM imported_data.diplomas;
   -- ... restore each table
   ```

5. **Verify integrity**:
   ```sql
   SELECT COUNT(*) FROM usuario;
   SELECT COUNT(*) FROM diplomas;
   -- Compare with backup_log.tamanho_bytes
   ```

## Troubleshooting

### Backup Job Failures

**Check logs**:
```bash
# Vercel deployment logs
vercel logs --since 1d

# Supabase function logs
# Dashboard тЖТ Functions тЖТ backup-criptografado тЖТ Logs
```

**Common issues**:

1. **Missing BACKUP_ENCRYPTION_KEY**
   - Error: "Missing required environment variables"
   - Solution: Add key to Vercel environment variables

2. **Database connection timeout**
   - Error: "Failed to export table"
   - Solution: Check Supabase service role key has database access

3. **Storage quota exceeded**
   - Error: "Upload failed"
   - Solution: Delete old backups or upgrade storage plan

### ZAP Scan Issues

**Scan times out**:
- Increase timeout in workflow (`timeout-minutes: 60`)
- Check if app is responding to requests

**False positives**:
- Add rule to allowlist: `.zap/allowlist.conf`
- Change rule action in `.zap/rules.tsv` to IGNORE/WARN

**Scan blocked**:
- ZAP IP rate-limited: Check error logs
- Solution: Configure rate limiting exceptions for GitHub Actions IPs

### GitHub Actions Failures

**Workflow doesn't run**:
- Check branch protection rules
- Verify `.github/workflows/*.yml` syntax: `yamllint`

**Secrets not available**:
- Confirm set in: Settings тЖТ Secrets and variables тЖТ Actions
- Check secret names exactly match

**Artifact upload fails**:
- Verify file exists at path specified
- Check artifact size < 5GB limit

## Security Best Practices

### For Backups

1. **Rotate encryption key** annually
   - Create new key: `openssl rand -base64 48`
   - Update `BACKUP_ENCRYPTION_KEY` in Vercel
   - Transition existing backups over time

2. **Test recovery** quarterly
   - Download encrypted backup
   - Decrypt and verify data integrity
   - Document recovery time estimate

3. **Monitor backup failures**
   - Set up alerts for status = 'erro'
   - Review logs weekly
   - Escalate repeated failures

4. **Backup retention policy**
   - Keep: Last 30 daily, 12 weekly, 6 monthly
   - Delete: Older backups automatically
   - Archive: Critical monthly backups offline

### For Security Scanning

1. **Review scan reports** weekly
   - Look for new vulnerabilities
   - Address FAIL rules before merge
   - Track WARN trends

2. **Update dependencies** regularly
   - Run `npm update` monthly
   - Review dependency changelogs
   - Test before deploying

3. **Fix vulnerabilities promptly**
   - Critical: within 24 hours
   - High: within 7 days
   - Medium: within 30 days
   - Low: planned update cycle

4. **Document exceptions** for ignored rules
   - Explain why rule is ignored
   - Review exceptions quarterly
   - Remove when no longer needed
