## Decisões arquiteturais tomadas autonomamente — **⚠️ PENDENTE APROVAÇÃO MARCELO**

Briefing S7 original deu espaço para decisões de implementação. Tomei 6 que divergem sutilmente do briefing literal e precisam de sua validação antes de virarem canônicas:

### D-S07-01 · RRF no cliente TS, não no banco
- **Briefing sugeria:** RRF em `retrieval/rrf.ts` do pacote (TS)
- **O que fiz:** mesma coisa, mas deixei explícito que o banco devolve scores brutos (`dense_score`, `sparse_score`) e o RRF acontece em memória no cliente
- **Trade-off:** mais testável, mais flexível (pesos ajustáveis por call), mas o banco manda mais dados pela rede
- **Alternativa:** UDF Postgres com RRF compilado. Ganha latência, perde flexibilidade
- **Status:** 🟡 PENDENTE

### D-S07-02 · Entity-boost via regex puro, sem LLM
- **Briefing sugeria:** "via LLM Haiku OU regex simples" — deixou escolha em aberto
- **O que fiz:** só regex (CPF/CNPJ/datas/R$/emails/nomes próprios). Zero custo, zero latência adicional
- **Motivo:** contexto Marcelo é financeiro/jurídico. Regex cobre 80% dos casos sem custo. LLM Haiku pode ser upgrade futuro
- **Custo de reverter:** baixo — interface `EmbeddingProvider` permite swap fácil
- **Status:** 🟡 PENDENTE

### D-S07-03 · Adapter para Art. XXII em vez de trocar o default stub
- **Briefing do S7 sugeria:** "Hook Art. XXII (já em S1, stub) passa a chamar `memory.add()` real"
- **O que fiz:** não mexi no pacote `constitutional-hooks`. Criei `createArtXXIIMemoryAdapter(memory)` em `@ecossistema/memory`. Consumer compõe no bootstrap
- **Motivo:** trocar o default em constitutional-hooks criaria dep `constitutional-hooks → memory`, e S10/S11 já dependem de ambos → ciclo virtual. O adapter inverte: memory fornece a ponte, hooks ficam zero-deps
- **Status:** 🟡 PENDENTE (alternativa: mudar constitutional-hooks para aceitar memory como peerDep opcional)

### D-S07-04 · Construtor dual — `MemoryConfig` OR `DirectDeps`
- **Briefing sugeria:** só `MemoryConfig` (URL + key)
- **O que fiz:** construtor aceita as duas formas. `DirectDeps` injeta `SupabaseClient` + `EmbeddingProvider` já construídos. Serve para testes e para consumers que querem reusar conexão
- **Motivo:** resolveu erro de tipagem em `withClient` estático; simplificou helper de testes
- **Status:** 🟡 PENDENTE

### D-S07-05 · Não aplicar outras 3 migrations da S4
- **Briefing S7 declara dependência de S4 (migrations 3-tier) como "já aplicadas"**
- **Realidade no banco:** só as 3 tabelas `memory_*` já existiam (aplicadas upstream em sessão anterior, sem registro local). `audit_log`, `skills_registry`, `ecosystem_credentials v2` também já estavam prontas
- **O que fiz:** adicionei APENAS 4 RPCs SQL necessárias ao cliente. Não toquei nas demais migrations da S4
- **Motivo:** migrations de S4 são domínio de outra sessão. Sobrepor seria scope creep
- **Status:** 🟢 OK (decisão defensiva, não canônica)

### D-S07-06 · Dimensão de embedding = 768 (confirmar consistência)
- **Schema pgvector:** `vector(768)` nas 4 colunas (`summary_vec`, `detail_vec`, `nl_vec`, `desc_vec`)
- **Gemini API:** `gemini-embedding-001` com `outputDimensionality: 768` (default seria 3072)
- **Status:** 🟢 OK — bate com §32 do Masterplan, mas vale reconfirmar se 768 é o que RAG-engine Railway usa também (senão embeddings inseridos por agentes e por RAG-engine ficam incompatíveis)
